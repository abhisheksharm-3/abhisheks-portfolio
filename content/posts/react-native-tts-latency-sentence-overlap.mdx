I was building a voice response loop for an offline React Native AI app, where the model responds and the device speaks it aloud, like a hands-free assistant. The first working version had a 3-4 second gap between the model finishing its response and the first word coming out of the speaker. On a Xiaomi running Android 15, synthesising a 120-word response with Kokoro took about 3.2 seconds before a single sample played.

The fix is overlap synthesis: split the response into sentences, start playing sentence 1 while sentence 2 is still being synthesised. The user hears the first word in under 500ms. The rest streams in behind it.

## The problem with synthesising the full response

Kokoro runs on-device via `react-native-executorch`. `forward()` takes a text string and returns a `Float32Array` of audio samples at 24kHz. The naive implementation looks like this:

```typescript
async function speak(text: string) {
  const samples = await tts.forward({ text, speed: 1.0 })
  await playSamples(audioCtx, toFloat32(samples))
}
```

For a 150-word response, `forward()` takes 2-4 seconds on mid-range hardware. The user waits in silence, then hears everything at once. This is fine for a chat interface. For a voice loop, that latency kills the experience.

The response already exists as text the moment the LLM finishes streaming. The only delay is synthesis. Synthesis is serial by default because `forward()` is one call per text block. But the response is naturally structured as sentences, and the device can play sentence 1 while synthesising sentence 2.

## Why forward() not stream()

Kokoro also exposes `stream()`, which sounds like the right tool. It isn't, on Android.

`stream()` uses a `for-await` loop over an async iterator. When you call `streamStop(true)` to cancel mid-response, the native side kills the generation thread but never signals the JS iterator as done. The `for-await` loop hangs indefinitely. `isGenerating` stays `true`. The next `speak()` call sees `isGenerating: true`, hits the guard check, and bails before synthesising anything. The device goes silent.

`forward()` resets `isGenerating` in its own `finally` block, whether it completes or throws. Cancellation is safe: poll `isGenerating` until it resets, then start the next `speak()` call.

```typescript
// Wait for any in-flight forward() to finish before starting a new speak().
// 3 seconds is a generous bound for a single sentence.
if (ttsRef.current.isGenerating) {
  const deadline = Date.now() + 3000
  while (ttsRef.current.isGenerating && Date.now() < deadline) {
    await new Promise<void>(r => setTimeout(r, 20))
  }
}
```

## The overlap pattern: drainQueue

The overlap pattern uses two concurrent loops. The synthesis loop iterates over sentences and calls `forward()` on each. The drain loop pulls chunks off a shared queue and plays them sequentially. They run concurrently: synthesis pushes, drain consumes.

```typescript
const playQueueRef = useRef<Float32Array[]>([])
const drainHolder: { p: Promise<void> | null } = { p: null }

// Drain loop: plays chunks in order, runs concurrently with synthesis.
const drainQueue = async (): Promise<void> => {
  while (playQueueRef.current.length > 0 && isMine() && !isStoppedRef.current) {
    const samples = playQueueRef.current.shift()!
    await playSamples(ctx, samples)
  }
}

// Synthesis loop: one forward() per sentence.
for (const sentence of sentences) {
  if (!isMine() || isStoppedRef.current) break

  const raw = await ttsRef.current.forward({ text: sentence, speed })
  const samples = trimTrailingSilence(toJsFloat32Array(raw as unknown as Float32Array))

  playQueueRef.current.push(samples)

  // Start the drain on the first chunk; subsequent chunks slot in behind it.
  if (!drainHolder.p) {
    drainHolder.p = drainQueue().finally(() => { drainHolder.p = null })
  }
}

// All sentences synthesised — wait for the queue to drain before returning.
if (drainHolder.p) await drainHolder.p.catch(() => {})
```

The drain starts as soon as the first sentence finishes synthesis. While sentence 1 plays (which takes however long its audio is), sentence 2 is synthesising. If sentence 2 finishes synthesis before sentence 1 finishes playing, it sits in the queue. If sentence 1 finishes before sentence 2 synthesises, there is a brief gap. In practice, a sentence long enough to need synthesis is also long enough to cover the next synthesis call.

**Cancellation** uses an ownership counter. Each `speak()` increments `speakIdRef` and captures the current value. Every `await` in the loop checks whether the captured value still matches:

```typescript
const myId = ++speakIdRef.current
const isMine = () => speakIdRef.current === myId
```

When a new `speak()` arrives mid-loop, it increments the counter. All `isMine()` checks in the old loop return `false` and the old loop exits at the next check point. No explicit cancellation signal needed. the counter is the signal.

## Sentence splitting and merging

`splitIntoSentences()` splits on `!`, `?`, and `;` boundaries. Short sentences (under 50 characters) get merged with the next one. The threshold exists because very short sentences (`"He was kind."` at 14 chars) produce audio that finishes before the next `forward()` call returns, creating an audible gap.

```typescript
const MIN_MERGE_LEN = 50
const sentences: string[] = []
let buf = ''

for (const s of rawSentences) {
  buf = buf
    ? `${buf.replace(/[;!?,.]+$/, ',')} ${s}`
    : s
  if (buf.length >= MIN_MERGE_LEN) {
    sentences.push(buf)
    buf = ''
  }
}
if (buf) sentences.push(buf)
```

Merge boundaries use a comma, not a semicolon or period. A comma produces the shortest natural pause Kokoro generates. shorter than a semicolon, longer than nothing. Merging with no punctuation produces flat prosody because the phonemiser has no breath cue between clauses.

## Text preprocessing for the phonemiser

Kokoro's phonemiser (espeak-ng) has opinions about punctuation. Several of them produce audio artefacts that make the voice loop sound wrong.

**Periods produce the longest pause.** A sentence-ending period is longer than a semicolon. Replacing `. ` before an uppercase letter with `; ` gives a more natural inter-sentence break:

```typescript
// ". " before uppercase → "; " — shorter, more natural pause
result = result.replace(/\.\s+(?=[A-Z])/g, '; ')
```

Run abbreviation expansion before this rule. `Dr. Smith` must become `Dr Smith` first, otherwise `Dr.` triggers the rule and produces `Dr; Smith`.

**Commas trigger a long clause pause.** espeak-ng inserts a noticeable pause for every comma followed by whitespace. Removing them keeps speech flowing:

```typescript
// Remove commas before whitespace — long pause artefact in espeak-ng
result = result.replace(/,(\s)/g, '$1')
// Numbers are safe: "1,000" has no trailing space
```

**Single quotes swallow words.** Kokoro's phonemiser treats text after an opening single quote as a phoneme escape. `'Sun God'` gets read as `God`. Strip surrounding quote marks from enclosed phrases. preserve contractions:

```typescript
// Strip surrounding quotes from phrases, preserve contractions like "it's"
result = result.replace(
  /['''""“”]([^'''""“”]+)['''""“”]/g,
  '$1'
)
```

**Newlines produce paragraph-length silence.** The phonemiser treats `\n` as a paragraph break. Collapse all newlines to spaces.

## Android quirks worth knowing

**`onEnded` does not reliably fire.** `AudioBufferSourceNode.onEnded` sometimes never fires on Android when a source finishes playing. Without a fallback, `playSamples()` never resolves and the queue stalls. Add a timeout based on audio duration:

```typescript
const durationMs = (samples.length / 24000) * 1000
// 300 ms grace period covers Android onEnded timing variance
const timeout = setTimeout(() => { cleanup(); resolve() }, durationMs + 300)
source.onEnded = () => { clearTimeout(timeout); cleanup(); resolve() }
```

**`isGenerating` resets asynchronously.** `forward()`'s `finally` block calls `setIsGenerating(false)`, which is a React state update. The state update is asynchronous. the next call to `ttsRef.current.isGenerating` may still see `true` for a frame or two after `forward()` resolves. Poll with a 500ms deadline before the next `forward()`:

```typescript
if (ttsRef.current.isGenerating) {
  const settle = Date.now() + 500
  while (ttsRef.current.isGenerating && Date.now() < settle) {
    await new Promise<void>(r => setTimeout(r, 10))
  }
}
```

**JSI-backed Float32Arrays lack a real ArrayBuffer.** `forward()` returns a JSI host object on Android. Calling `buffer.copyToChannel(raw, 0)` throws `"getPropertyAsObject: property 'buffer' is undefined"` because JSI host objects do not have a standard `ArrayBuffer`. Copy to a JS-heap `Float32Array` first:

```typescript
function toJsFloat32Array(src: Float32Array): Float32Array {
  const dst = new Float32Array(src.length)
  dst.set(src)
  return dst
}
```

**Trim trailing silence.** Kokoro appends silence at the end of each synthesised chunk. Without trimming, every sentence ends with a dead pause before the next one begins:

```typescript
const SILENCE_THRESHOLD = 0.001
const MIN_TRIM_SAMPLES = 1200

function trimTrailingSilence(chunk: Float32Array): Float32Array {
  let end = chunk.length - 1
  while (end > 0 && Math.abs(chunk[end]) < SILENCE_THRESHOLD) end--
  const trimmed = chunk.length - 1 - end
  // Only trim if the silence is long enough to be audible (~50ms at 24kHz)
  return trimmed > MIN_TRIM_SAMPLES ? chunk.slice(0, end + 1) : chunk
}
```

The combination of sentence overlap, trailing silence trimming, and the 300ms `onEnded` grace period brings perceived response latency from 3-4 seconds to under 500ms for the first word on the devices I tested.
