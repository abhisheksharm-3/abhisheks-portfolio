I was building a hands-free conversational mode for an offline React Native AI app. The loop was: user speaks, on-device model responds, TTS plays the response, user can interrupt mid-response to redirect. The last piece was interruption detection. I used RMS amplitude on the microphone input with a threshold of 0.02. It worked on the first manual test. It failed on every automated run after that.

The assistant was interrupting itself. Every TTS sentence would cross the amplitude threshold, fire the interrupt handler, stop playback, and start listening for new input. The new input was silence, because the user hadn't spoken. The session would spin in a loop of interrupted half-sentences.

The root cause took about 30 minutes to find, and the fix was 1 line.

## What rms detection actually measures

RMS (root mean square) amplitude is a measure of the average energy in an audio buffer. The formula is straightforward:

```typescript
function rms(buffer: Float32Array): number {
  const sumOfSquares = buffer.reduce((sum, sample) => sum + sample * sample, 0)
  return Math.sqrt(sumOfSquares / buffer.length)
}
```

You call this on each audio frame from the microphone and compare against a threshold. Values above the threshold mean audio is present. Values below mean silence.

The problem is what "the microphone" means when you haven't specified an audio source.

## Why the microphone hears your tts

On Android, the default audio source (`AudioSource.DEFAULT`, also equivalent to `AudioSource.MIC`) gives you raw microphone input. Raw means everything the physical microphone transducer picks up, including sound waves that originated from the device's own speaker.

On a phone held in hand, speaker audio reaches the microphone at around -20 to -30 dB relative to the original signal. At moderate TTS volume, that's still well above any practical interrupt threshold. The microphone hears the speaker playing, your RMS check sees amplitude, and the interrupt fires.

This is not a threshold calibration problem. Lowering the threshold would make it miss genuine user speech. Raising it would require the user to shout. There's no value that distinguishes "user speaking" from "device speaker audio reaching the microphone" using raw mic input.

<PostCallout>
The same failure mode affects any voice activity detection (VAD) implementation that runs on raw microphone input during audio playback. WebRTC VAD, silero-vad, or any energy-based detection will false-positive during TTS unless the speaker signal is removed first.
</PostCallout>

## The fix: AudioSource.VOICE_COMMUNICATION

Android provides `AudioSource.VOICE_COMMUNICATION` (integer value 7) specifically for two-way voice applications. When you select this source, Android activates the device's hardware Acoustic Echo Cancellation processor before the audio stream reaches your app.

Hardware AEC runs in device firmware. It maintains a reference signal of what the speaker is currently playing and subtracts that signal from the microphone input in real time. What your app receives is microphone audio with the speaker echo removed. The user's voice is preserved. The TTS playback audio is not.

The change in expo-av:

```typescript
// Before: raw mic input, picks up speaker
await recording.prepareToRecordAsync({
  android: {
    extension: ".wav",
    outputFormat: Audio.AndroidOutputFormat.DEFAULT,
    audioEncoder: Audio.AndroidAudioEncoder.DEFAULT,
    sampleRate: 16000,
    numberOfChannels: 1,
    bitRate: 128000,
  },
  ios: { /* ... */ },
  web: {},
})

// After: hardware AEC enabled, speaker audio removed
await recording.prepareToRecordAsync({
  android: {
    extension: ".wav",
    outputFormat: Audio.AndroidOutputFormat.DEFAULT,
    audioEncoder: Audio.AndroidAudioEncoder.DEFAULT,
    sampleRate: 16000,
    numberOfChannels: 1,
    bitRate: 128000,
    audioSource: 7, // AudioSource.VOICE_COMMUNICATION
  },
  ios: { /* ... */ },
  web: {},
})
```

expo-av doesn't export an enum for Android audio source values, so you pass the integer directly. `7` is `MediaRecorder.AudioSource.VOICE_COMMUNICATION` from the Android SDK.

## The implementation

With hardware AEC active, the interrupt detector becomes a straightforward speech detector rather than an echo filter. The threshold check now answers "is the user speaking" rather than "is someone or something making noise":

```typescript
const INTERRUPT_THRESHOLD = 0.015
const SUSTAINED_FRAMES_REQUIRED = 3

let sustainedFrames = 0

function onAudioFrame(buffer: Float32Array): void {
  const amplitude = rms(buffer)

  if (amplitude > INTERRUPT_THRESHOLD) {
    sustainedFrames++
    if (sustainedFrames >= SUSTAINED_FRAMES_REQUIRED) {
      handleInterrupt()
      sustainedFrames = 0
    }
  } else {
    sustainedFrames = 0
  }
}
```

The sustained-frames check requires 3 consecutive frames above threshold before firing. This filters out brief spikes from ambient noise (keyboard clicks, notification sounds, door closing in the background). With 16kHz sample rate and 1024-sample frames, 3 frames is about 192ms. Long enough to reject transients, short enough to feel responsive.

The `handleInterrupt` function stops TTS playback, cancels any in-flight LLM request if possible, and starts the ASR transcription pass on whatever the user just said:

```typescript
async function handleInterrupt(): Promise<void> {
  // Stop TTS immediately
  await ttsEngine.stop()

  // Cancel in-flight inference if the model supports it
  if (currentInferenceController) {
    currentInferenceController.abort()
  }

  // Start transcribing the captured audio
  const transcription = await transcribeAudio(capturedAudioBuffer)
  if (transcription.trim().length > 0) {
    await processUserInput(transcription)
  }
}
```

One detail: you need to be recording continuously while TTS plays, not only after it ends. Start the recording session before the TTS response begins and keep it running through playback. The hardware AEC handles removing the speaker audio; your job is to keep the buffer populated so you can analyze it in real time.

## iOS and the audio session mode

iOS doesn't expose audio source types the same way Android does. The equivalent is setting the `AVAudioSession` mode to `.voiceChat` before starting the recording session.

In React Native, this requires configuring the audio session explicitly:

```typescript
await Audio.setAudioModeAsync({
  allowsRecordingIOS: true,
  playsInSilentModeIOS: true,
  staysActiveInBackground: false,
  // .voiceChat mode enables echo cancellation on iOS
  iosCategory: "playAndRecord",
  iosCategoryMode: "voiceChat",
  iosCategoryOptions: [
    "defaultToSpeaker",
    "allowBluetooth",
  ],
})
```

The `iosCategoryMode: "voiceChat"` tells iOS to activate its echo cancellation chain. The behavior is similar to Android: the system removes the speaker signal from microphone input before your app sees it.

One difference: iOS `.voiceChat` mode also changes the audio output routing toward the earpiece by default, which may not be what you want for a device sitting on a desk. Override with `defaultToSpeaker` in the category options to keep audio coming from the main speaker while still getting echo cancellation on the input.

## What to keep after the fix

After switching to hardware AEC, I removed:
- The adaptive threshold logic that tried to track background noise levels
- The cooldown timer that blocked interrupts for 500ms after TTS started (to "skip" the initial speaker bleed)
- The gain normalization pass that was trying to compensate for mic-to-speaker distance variation

All of those were patches for the wrong problem. With hardware AEC, the microphone input is clean and a fixed threshold works correctly.

The one thing worth keeping: the sustained-frames check. Hardware AEC removes the echo but not ambient room noise. Brief spikes from non-speech audio still appear. Requiring 3 consecutive frames above threshold eliminates almost all false positives from environmental noise without adding perceptible response latency.

The entire fix was changing one integer in the recording config. Everything else was removing code that was compensating for the wrong audio source.
